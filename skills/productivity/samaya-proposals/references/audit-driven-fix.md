# Audit-Driven Proposal Fix Workflow

When the user says "use <agent> to fix the proposal based on its audit report":

## Phase 1 — Locate Files

1. Find the proposal HTML (could be in /tmp/, OneDrive, or project folder)
2. Find the audit report (generated by Kimi or another agent as a .md file)
3. Read both fully to understand current state vs. required fixes

## Phase 2 — Pre-flight Check

**Before applying any fix, check which issues are already resolved.** Audit reports often flag items that were already fixed in a prior round but the audit was run against an older snapshot.

Common false-positive audit items:
- "Exhibition area ~1,500 m² → should be ~3,500+ sqm" — may already be corrected
- "Schedule 36 weeks → should be 52 weeks" — may already be 52 weeks
- "SOW/ER references not cleaned" — may already be cleaned from the proposal body but still present in appendices
- "Available on request (CVs)" — may have already been changed to "attached in Package B"
- "Only 14/50 compliance items detailed" — may have all 50 items already

Use `search_files(output_mode="count", pattern="<old_text>")` on the HTML to verify each finding's current state. Only patch things that are actually wrong.

## Phase 3 — Delegate Mechanism

**Prefer `delegate_task` over raw CLI invocation.** The raw `kimi -p "..."` with PTY mode regularly hits the 600s timeout on files >150KB with 22+ fixes.

### delegate_task approach (preferred)
```python
# Create a sub-agent with terminal+file tools for HTML patch work
delegate_task(
    goal="...",
    context="...",
    toolsets=["terminal", "file"]
)
```

Advantages:
- Separate timeout (not bounded by the parent's 600s)
- Sub-agent has read_file, patch, search_files, terminal — all the tools needed for multi-fix HTML surgery
- Sub-agent returns a structured summary of what was done
- Sub-agent can iterate through patch() calls without the parent's iteration limit

### Raw kimi CLI approach (fallback only)
```bash
kimi -p "Fix all issues..." --max-steps-per-turn 50 --print -y
```
Limitations:
- PTY mode + large file + many fixes = ~90% timeout rate
- No way to checkpoint progress
- Output is a wall of text, hard to verify

## Phase 4 — Fix Application Pattern

The sub-agent should:
1. Read both files (proposal + audit report)
2. Verify each finding's current state with search_files
3. Apply fixes in order: critical → high → medium → low
4. Use patch() for targeted string replacements (not write_file which replaces everything)
5. Use one patch() call per atomic fix
6. Verify each patch succeeded (patch() returns a diff)
7. After all patches, run a structural check (tag balance, page count, key terms)

## Phase 5 — Verification

After the sub-agent returns, verify key fixes in the parent turn:

```python
checks = {
    'old area': '~1,500 m²' not in content,
    'old schedule': '36 weeks' not in content,
    'CVs note': 'available on request' not in content,
    'new appendix': 'Appendix F' in content,
    'expanded risks': 'R-20' in content,
    'correct page count': '22 / 22' in content,
}
```

Check each. If any fail, investigate — the sub-agent may have missed it.

## Pattern: "Fixes Already Applied" Discovery

A sub-agent with full tool access can discover that some audit findings are stale (already fixed in a prior rebuild). This is common when:
- The proposal went through multiple rebuild iterations
- Different sections were fixed at different times
- The audit was run against an intermediate version

The sub-agent should explicitly report which fixes were "already applied" vs "newly applied" vs "skipped (not applicable)". This prevents redundant changes and wasted effort.

## Known Pitfalls

- **Raw CLI timeout** — kimi PTY mode on HTML >150KB with 22+ fixes times out ~90% of the time. Always use delegate_task for this work.
- **Stale audit findings** — Always verify before patching. A finding that says "SOW/ER references still present" may have been fixed since the audit was run.
- **File not found** — The proposal may exist in /tmp/ but not in the OneDrive path. Check /tmp/ early.
- **Patch failure on fuzzy match** — If the proposal was rebuilt after the audit, exact strings from the audit report may not match. The sub-agent needs to search for functionally equivalent text, not just the exact string from the audit.
- **Page count drift** — After adding Appendix E/F and milestone tables, the page count changes. The autopaginate script at the bottom of the HTML must be updated.
