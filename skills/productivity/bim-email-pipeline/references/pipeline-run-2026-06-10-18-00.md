## Arabic Filename Normalization Issue — Sub-Agent False Positive

The Codex sub-agent reported 4 "new" files (general/ category) that were "not found in any BIM folder." Closer inspection revealed they **were already present** in BIM OneDrive — the sub-agent's comparison failed because Arabic characters had different Unicode normalization forms:

| Sub-agent report | Actual BIM file | Issue |
|-----------------|-----------------|-------|
| `35- العرض الفني والمالي - شركة سمايا الإستثمارية - أعمال الدراسات والتصميم الإنشائي متحف عسير.pdf` | `...الإستثمارية - أعمال الدراسات والتصميم الإنشائي متحف عسير.pdf` | `ا` (U+0627) vs `أ` (U+0623); `ي` (U+064A) vs `ى` (U+0649) |
| `AL JALAL WAL JAMAL` files | Same files in `99_Images/` and `Docs/00_Admin/` | Exact match but find scope missed them — the sub-agent only searched 7 BIM folders, not the full tree |

**Lesson:** When delegating file comparison to sub-agents for BIM projects with Arabic names:
1. Pass the full BIM tree `find -iname` results, not just 7 specific folders
2. Normalize filenames before comparison: `unicodedata.normalize('NFC', name)` and lowercase
3. Strip Arabic diacritics (tashkeel) before comparing: `re.sub(r'[\u064B-\u0652]', '', name)`
4. Consider using `difflib.SequenceMatcher` with ratio > 0.9 for Arabic filenames instead of exact match
