# Risk Management Plan — DOCX Patterns

## Document structure (CG-ready)

| Section | Content | Status |
|---------|---------|--------|
| Cover | Doc ref, contract, employer, contractor, PMC, design lead, location, revision, date, classification | First page |
| DC block | QC table: Prepared by / Reviewed by / Approved by (actual names, not roles) | First page |
| Revision history | Version, Date, Author, Changes. REV00 = "First issue for CG review" | First page |
| TOC | Word TOC field (`TOC \o "1-2" \h \z \u`), update in Word | After first page |
| Sections 1-12 | Methodology (how risks are managed) | Body |
| Section 14 | Register Status Summary | Body |
| C Abbreviations | Standard | End |
| Document Control | Revision history table | End |

## Sections to remove for CG submittal

- Section 13 — Integration with Other Plans (if referenced plans don't exist)
- Section 15 — Appendices (empty header)
- Appendix A — Risk Register Location (internal tracking)
- Appendix B — Key Risk-Related Documents and Data Gaps (internal tracking)

## Severity scale — project-aligned thresholds

For a project with ~10 weeks remaining (e.g. 30 Sep 2026 deadline):

| Rating | Schedule Impact |
|--------|----------------|
| Low | Less than 1 week |
| Medium | 1 - 2 weeks |
| High | 2 - 4 weeks |
| Very High | More than 4 weeks |

## Live register notes

Add halftone (9pt `#64748B`) note after every table showing register data:
> "Data shown is a snapshot from the live Project Risk Register, which is the authoritative source and updated weekly."

Tables that need it:
- 2.1 Current Risk Snapshot
- 4.2 Current Risk Distribution
- 14 Register Status Summary

## RBS empty categories note

Add after the RBS distribution table:
> "Note: RBS categories with no active risks are monitored and will be populated as new risks emerge from design development, procurement, and site activities."

## First-page layout order

Cover table → DC block → Revision history → TOC → document body. Add spacing paragraphs between tables.

## CG response forecast

A methodology-only risk plan (describes HOW, not the specific risks) should forecast **Code A** if:
- No TBCs or placeholders in methodology sections
- No internal file paths or repo references
- Empty RBS categories explained with a monitoring note
- AV register "in progress" noted as factual (register tracks progress)
- DC block has actual names, not role titles
- Professional formatting (logo, header, footer, TOC)
- Schedule thresholds proportional to project duration

## REV00 entry

Client-appropriate, not a changelog:
- ✅ "REV00 - First issue for CG review"
- ❌ "Format revision — unified table styles, halftone remarks, page breaks, removed internal references"
