### Aseer Email_Archive MD Filename Parsing (Primary Cron Source)

The `Email_Archive/` directory under `Aseer-Museum/` contains timestamped markdown files with the format:
```
YYYY-MM-DD - <Doc Description>.md
```

**How to parse** (use regex on filename, not file content — filenames are always readable even when OneDrive-locked):

```python
import re, os
from datetime import datetime, timedelta

AS_ARCHIVE = "/Users/mohamedessa/Library/CloudStorage/.../Aseer-Museum/Email_Archive"
cutoff = (datetime.now() - timedelta(days=7)).date()

# Generated files (inventories, archives, mappings) — always skip
SKIP_FILES = {'CG_comprehensive_document_status_mapping.md',
              'hesham_archive_may2026.md', 'cg_response_docs_inventory.json',
              '_aseer_tasks_backlog.md', '_email_processing_log.md'}
# Doc codes whose content type is encoded in the prefix — used below
DOC_PREFIXES = ('zd-', 'pl-', 'sc-', 'pq-', 'sh-', 'ir-', 'rp-', 'fr-',
                'wf-', 'fl-', 'pr-', 'skr-', 'ifc-', 'si-', 'tq-', 'ma-')

for fname in os.listdir(AS_ARCHIVE):
    if not fname.endswith('.md'): continue
    if fname in SKIP_FILES: continue
    m = re.match(r'(\d{4}-\d{2}-\d{2})\s*[–\-]\s*(.+)\\.md$', fname)
    if not m: continue
    fdate_str, desc = m.group(1), m.group(2).strip()
    fdate = datetime.strptime(fdate_str, '%Y-%m-%d').date()
    if fdate < cutoff: continue
    # Classify by doc code in description
    d = desc.lower()
    # HIGH priority
    if 'ir-' in d or 'inspection request' in d or 'installation' in d:
        cat = 'IR'
    elif 'hse' in d or 'fire' in d or 'emergency' in d or 'safety' in d or 'security' in d or 'breach' in d or 'facp' in d or 'training matrix' in d or 'heat stress' in d or 'lifting operation' in d or 'mobile equipment' in d:
        cat = 'HSE'
    elif 'mep' in d or 'mechanical' in d or 'ميكانيكا' in desc or 'كهرباء' in desc:
        cat = 'MEP'
    elif 'طلب عرض سعر' in desc or 'rfp' in d or 'rfq' in d or 'proposal' in d or 'quotation' in d or 'pricing' in d:
        cat = 'RFP'
    elif 'mobilization' in d:
        cat = 'MOB'
    elif 'si-' in d or 'site instruction' in d or 'jsi' in d or 'joint site instruction' in d:
        cat = 'SI'
    # MEDIUM priority
    elif 'pq-' in d or 'vendor' in d or 'projector' in d or 'panasonic' in d:
        cat = 'PQ'
    elif 'sh-' in d or 'جدول' in desc or 'schedule' in d:
        cat = 'SCH'
    elif 'pl-' in d or 'zd-' in d or 'sc-' in d or 'management plan' in d or 'لائحة' in desc or 'خطة' in desc:
        cat = 'DOC'
    # LOW priority
    elif 'rp-' in d or 'report' in d or 'assessment' in d:
        cat = 'RP'
    elif 'طلب سيارة' in desc or 'سيارة' in desc:
        cat = 'CAR'
    else:
        cat = 'GEN'
    add('Aseer', desc, cat, fdate_str)
```

### Critical SDR Submittal Keywords (elevate to HIGH)
- HSE, FIRE, EMERGENCY, FIRE PREVENTION, SAFETY, EVACUATION, BREACH, FACP

### Code C Elevation Rule
Items returned from the consultant with **Code C (Revise & Resubmit)** are HIGHER priority than their category would suggest:
- A Code C item carries a hard deadline (typically 7-14 days).
- Example: PL-0057 Time Schedule (SCH→MEDIUM) with Code C → elevate to **HIGH**.
- Check `_aseer_tasks_backlog.md` "Critical Items Needing Action" table for explicit deadlines.
- Cross-reference with `_Project_Memory/WEEK*_UPDATE.md` for actual status change confirmations.

### SI Priority: Security/HS Keywords Elevate
SI/JSI items are 🟡 MEDIUM by default. Elevate to 🔴 HIGH if the description contains: `security`, `breach`, `fire`, `facp`, `emergency`, `safety`, `power disconnection`. These are critical site safety/security instructions requiring immediate attention.

## Filters

- **Date range**: last 7 days from today
- **Status**: only `Active` or `Open` entries — exclude `Archived`
- **Deduplication**: by first 50 characters of subject line (merge threads spanning multiple dates)
- **Max items**: 15 total — if more, summarize lowest-priority items as "Plus N more routine reviews"

## Open-Entry Format (NRS New Entries)

`Open` entries have a different format from `Active` email entries:
- `Ref` is a numeric ID (e.g., `29398`) instead of `ASE-EMAIL-NNNNN`
- `Action` field is the action verb (e.g., `Review`, `Process Payment`) instead of `Reviewed / Reply Sent`
- Subject line is the actual task description, not an email subject
- They are high-value actionable items — always include them in the output

Example:
```
2026-05-28,GEN,29400,Review,Invoice NRS-SAM-2026-004 (INV-4825),Process Payment
```

### Parsing Approach

Markdown files use a consistent format:
```markdown
### N. YYYY-MM-DD — Category Name
**File:** <filename>
**Doc Code:** `<CODE>`
```

Example:
```markdown
### 48. 2026-05-21 — تقديم مستندات
**File:** 2026-05-21 - RE  MOC-MUS-ASE-1KH-PL-0037   Workers Welfare Management Plan.md
**Doc Code:** `MOC-MUS-ASE-1KH-PL-0037`
```

Parse with regex:
```python
import re
from datetime import datetime

entries = []
for line in content.split('\n'):
    entry_match = re.match(r'###\s+\d+\.\s+(\S+)\s+—', line)
    if entry_match:
        try:
            date = datetime.strptime(entry_match.group(1), '%Y-%m-%d')
        except ValueError:
            date = None
        entries.append({'date': date, 'line': line})
    doc_match = re.search(r'\*\*Doc Code:\*\*\s*`(.+?)`', line)
    if doc_match and entries:
        entries[-1]['doc_code'] = doc_match.group(1)
```

### Cross-Project File Detection (Watchdog)

Files from one project sometimes appear in another project's tracked folder. For example, during this session:
- **Zamzam Quotation** (`Quotation NO.3451- SAMAYA INVEST- ZAMZAM MUSEUM.pdf`) was found in the Aseer `00_Scope_and_Proposals/` folder (June 13)

Likewise, when scanning Zamzam, check for "Aseer" in Zamzam paths.

This cross-contamination happens because quotes, proposals, and reference materials are sometimes stored under the source project's folder rather than the destination's.

### Method — json.load (works, use this)

```python
import json
from datetime import datetime, timedelta

today = datetime.now()
cutoff_ts = (today - timedelta(days=7)).timestamp()

with open('/Users/mohamedessa/.hermes/scripts/.watchdog_state.json', 'r') as f:
    data = json.load(f)

zamzam_files = []
for path, info in data.items():
    if 'Zamzam' not in path:
        continue
    mtime = info.get('mtime', 0)
    if mtime < cutoff_ts:
        continue
    name = path.split('/')[-1]
    fdate = datetime.fromtimestamp(mtime).strftime('%Y-%m-%d')
    zamzam_files.append({'path': path, 'name': name, 'date': fdate, 'mtime': mtime})
```

### Cron Mode — execute_code Blocked

In cron mode, `execute_code()` sandbox has **subprocess blocked** — calls like `subprocess.run()` raise `BLOCKED: execute_code runs arbitrary local Python...`. Cannot pipe watchdog output through grep inside `execute_code()`.

**Alternative for cron:** Use `terminal()` with a heredoc `python3 << 'PYEOF' ... PYEOF` block, or write a temp `.py` file and execute with `terminal("python3 /tmp/script.py")`.

Example:
```bash
python3 << 'PYEOF'
import json, os
with open(os.path.expanduser('~/.hermes/scripts/.watchdog_state.json')) as f:
    data = json.load(f)
for path, info in data.items():
    if 'Zamzam' not in path: continue
    mtime = info.get('mtime', 0)
    if mtime >= 1781384400.0:

PYEOF
```

**Failure Mode A — File Does Not Exist:**
Either CSV may not exist locally if OneDrive never synced the directory. `find` returns nothing.

**Failure Mode B — OneDrive File Lock:**
Files that DO exist may return `Errno 11` when OneDrive holds an exclusive lock. Applies to both CSVs intermittently.

**Failure Mode A — File Does Not Exist:**
The paths `Docs/Email Archive/مشروع متحف عسير الإقليمي/Register_ASEER_Professional.csv` and `Docs/Email Archive/مشروع تأهيل مسار الزوار زمزم/Project_Log_مشروع تأهيل مسار الزوار زمزم.csv` may not exist at all on the local filesystem. The `Email_Archive/` directories under both projects are often empty (OneDrive Files On-Demand placeholders that were never synced). `find` returns nothing. Even the parent subdirectory structure may be absent. In this case, no workaround can read them — the bytes simply aren't there.

**Failure Mode B — OneDrive File Lock (Resource deadlock avoided):**
Files that DO exist return `Errno 11` on any read attempt because macOS OneDrive's sync engine holds an exclusive lock. This affects all read paths: `cat`, `head`, `dd`, Python `open()`, etc.

However, **`find -newermt` works** on locked stubs because it only checks filesystem metadata, not content:
```bash
find '/.../Aseer-Museum' -type f -newermt 'YYYY-MM-DD' 2>&1 | grep -vE '(\.DS_Store|backup|\.dat)' | head -40
```

Similarly, `ls -lt` works on parent directories of locked files — use it on `Email_Archive/Attachments/` and `Email_Archive/_attachments/` to see active PDF attachment names with dates.

**Diagnosis:**
- `ls -lO <path>` shows the `compressed,dataless` flag for Failure Mode B
- `file <path>` returns `ERROR: cannot read ... (Resource deadlock avoided)` or `empty`
- `find` returns nothing for Failure Mode A
- **`read_file` signal**: `total_lines=0` with non-zero `file_size` (e.g., `file_size: 356832, total_lines: 0`) means the file is **hydrated but locked** — bytes exist on disk but OneDrive holds an exclusive read lock. This is distinct from a dataless stub (0 bytes/not on disk), but both produce `Errno 11` on `terminal()` reads.
- Try: `python3 -c "with open('path') as f: print(len(f.read()))"` — if it errors, the file is locked or absent
- Use `mdls -name kMDItemDownloadedDate <path>` to confirm hydration status. Returns `(null)` when file is a dataless stub not yet synced locally. This is more reliable than `ls -lO` because it directly checks the file provider's download state rather than filesystem flags.
- Use `lsof <path>` to identify the PID holding the lock — this confirms whether OneDrive (CloudBrain) or a stray Python process is the culprit. A Python PID suggests a previous script run still holding the handle; `kill -9 <pid>` can free it, though in cron context it's safer to fall through to other data sources.

**Diagnosis pipeline (recommended order):**
```bash
# 1. Quick file existence check
ls -la "$path" 2>&1
# 2. Hydration state — is the file actually on disk?
mdls -name kMDItemDownloadedDate "$path" 2>/dev/null | grep -q '(null)' && echo "DATALESS STUB" || echo "HYDRATED"
# 3. Lock holder
lsof "$path" 2>/dev/null | head -3
# 4. Try read
python3 -c "with open('$path') as f: print(len(f.read()))" 2>&1
```

### OneDrive path contains non-ASCII Arabic characters
Paths may appear empty or return 0 lines when read via `read_file` — the file is fine, the path encoding differs. Use `terminal()` with explicit path quoting for CSV inspection.

### read_file adds line number prefixes (`N|`)
`read_file()` prepends line numbers with `|` separator, which breaks `csv.DictReader`. Always use the `terminal("cat ...")` + `io.StringIO(csv_text)` pattern instead. See `references/csv-handling.md`.

### Aseer CSV uses `\r\n` (DOS) line endings

The `Register_ASEER_Professional.csv` file uses Windows-style `\r\n` line endings. Python's `csv.DictReader` with `io.StringIO` handles these, but **only if** you use `.replace('\r\n', '\n')` when reading:

```python
with open(path, 'r') as f:
    reader = csv.DictReader(io.StringIO(f.read().replace('\r\n', '\n')))
```

Without this, the last field of each row gets a trailing `\r` — e.g., `Active\r` instead of `Active`, breaking `status == 'Active'` comparisons.

### `terminal("cat largefile")` truncates output

`terminal("cat ...")` has an output cap — for the watchdog JSON (~4.5 MB / 57K lines), it returns only ~330 lines. This silently produces incomplete data. **Do not** pipe large files through `terminal()`.

Use these alternatives instead:
- **In `execute_code()` sandbox**: read the file directly with Python `open()` — the sandbox has full filesystem access
- **Via temp script**: `write_file()` a `.py` script, then `terminal("python3 /tmp/script.py")` to execute it

### `python3 -c "..."` quoting breaks on complex logic

Inline `terminal("python3 -c 'print(json.load(f...))'")` fails when the Python code contains nested quotes, JSON strings, Unicode, or complex control flow. The shell's quoting rules conflict with Python's.

**Fix:** Write the script to a temp file first:

```python
write_file('/tmp/extract.py', '...python code...')
result = terminal('python3 /tmp/extract.py')
```

### Telegram output blocked by security scan — emoji variation selectors

When generating Telegram output with emoji characters like `🏗️` or `🕋`, Hermes' security scanner may flag the Unicode variation selectors (VS1-VS256, `U+FE0F`) that follow certain emoji. This blocks output entirely with a `[MEDIUM] Variation selector characters detected` error.

**Fix:** Use plain emoji without variation selector suffixes. In Python, strip the `\\ufe0f` character:
```python
emoji = "🏗️".replace("\\ufe0f", "")  # or just type 🏗 without the variation selector
```

Alternatively, avoid emoji entirely and use text markers (`[HIGH]`, `[MED]`, `[LOW]`). This is the safer choice for cron delivery because:
- Variation selector stripping is fragile across Python versions
- Telegram renders emoji fine without VS suffixes
- Text markers survive any future security scanner changes

Also test your output format by writing to a temp file (`write_file /tmp/out.txt`) before printing to stdout — this catches security blocks without losing work.

### Date filter must use date-only comparison to avoid time-component exclusion
Using `datetime.now() - timedelta(days=7)` produces a `datetime` with the current time (e.g., `May 24 05:06:00`). Comparing against `datetime.strptime(date, '%Y-%m-%d')` (which starts at midnight) causes entries from exactly 7 days ago to be silently excluded (`May 24 00:00:00 < May 24 05:06:00` → True → filtered OUT).

**Fix:** Use `.date()` objects for comparison:
```python
cutoff = (datetime.now() - timedelta(days=7)).date()  # date-only, no time
entry_date = datetime.strptime(date_str, '%Y-%m-%d').date()
if entry_date < cutoff:
    continue  # correctly excludes only genuinely older entries
```

### Fallback Workflow Steps

1. **Confirm registers and markdown summaries are unavailable** (Step 0 pre-flight already failed all three tiers). Proceed.
2. **Run `find -newermt` metadata scan** on both project directories for the last 7 days — this is faster and more targeted than scanning the full 4.3 MB watchdog JSON:
   ```bash
   find '/.../Aseer-Museum' -type f -newermt '7 days ago' 2>&1 | grep -v '.DS_Store' | head -40
   find '/.../Zamzam Museum' -type f -newermt '7 days ago' 2>&1 | grep -v '.DS_Store' | head -40
   ```
   This reveals recently modified files even on locked stubs. Pipe through `grep -v 'backup\|temp\|\.dat'` to filter noise.
3. **Extract recent files** from watchdog_state.json (last 7 or 14 days) using `json.load()` (preferred) or the line-by-line parser (fallback — see `references/watchdog-parsing.md`).
3. **Categorize files** by the pattern table above — group similar files (e.g., 67 furniture sheets → one "Furniture Shop Drawings" item).
4. **Identify missing registers** from the audit report — these become open tasks.
5. **Check for known open items** from the project index (stakeholders/summary sections may contain active SI numbers, DMP revisions, etc.).
6. **Apply deduplication** — collapse batch uploads of similar file types into a single todo item with count.
7. **Cap at 15 items** — batch-group the lowest-priority routine items.
8. **Include a caveat** at the top: "⚠️ Registers unavailable (OneDrive not synced) — TODO based on file activity + audit data."

- **Aseer Scope_and_Proposals/ folder**: Contains cross-project items — scan it for "Zamzam" strings when Aseer returns results, and vice versa. In June 2026, a Zamzam quotation (NO.3451) was found in `Aseer-Museum/Design Files/00_Scope_and_Proposals/`.

## Reference: Aseer Register CSV Format

```
Date,Ref,Category,From,To,Subject,Action,Remarks,Status
2026-05-25,ASE-EMAIL-34352,RFP,omar.bittar@3d-me.com,omar.bittar@3d-me.com,RE: RFQ — Artec Space Spider | Aseer Regional Museum — Replica Fabrication Project,Reviewed / Reply Sent,"Glad to have our meeting today...",Active
```

- Date format: `YYYY-MM-DD`
- Status `Active` = needs attention; `Archived` = done; `Open` = NRS new entry
- `Remarks` field often contains Arabic text (`عناية المهندس / محمد الباز المحترم`)
