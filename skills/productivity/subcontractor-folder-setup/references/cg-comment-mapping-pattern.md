# CG Code B Comment Mapping — Pattern

When a subcontractor's SOW is returned by CG with Code B (Approved with Comments), the comments must be mapped to existing scope sections to prepare the CRS (Comment Response Sheet).

## Standard CG Comment Categories

KSA museum project CG reviewers tend to issue the same ~9 comment types regardless of discipline:

| # | Comment Type | Typical Wording | Likely Covered? |
|---|-------------|-----------------|-----------------|
| 1 | **Cover all systems/spaces** | "Shall cover all engineering work for all systems/spaces including but not limited to: [list]" | Usually yes — sub's scope already lists these. May need to add missing line items. |
| 2 | **Full contractual compliance** | "Shall adhere with all requirements under complete project contractual package" | Always standard boilerplate. Respond: "Noted — confirmed." |
| 3 | **Action plan** | "Shall provide complete dated action plan for review and approval covering all activities under its SOW, including detailed submission plan" | Often not in sub's scope initially. Need to add as a deliverable. |
| 4 | **Coordinate with lead designer** | "All deliverables shall be fully coordinated with the lead designer [NAME] and other stakeholders" | Usually covered in sub's coordination section — may need to name the specific designer. |
| 5 | **Comply with DMP** | "Shall comply and cover all requirements at project DMP" | Standard — respond: "Noted — confirmed." |
| 6 | **All tasks reasonably required** | "Shall carry out all tasks required to successfully complete the project, whether specifically stated or reasonably required" | Standard contractual language — respond: "Noted — confirmed." |
| 7 | **Full coordination** | "Full coordination shall be maintained with the Client, PMC, Consultant, Contractor, and all related parties" | Usually covered in scope — respond: "Noted — confirmed." |
| 8 | **Meetings/site visits** | "Attend all required meetings, workshops, technical reviews, and site visits" | Sometimes not explicitly stated. Add as a commitment. |
| 9 | **Coordinate disciplines** | "Coordinate with Architectural, Structural, MEP, AV, Exhibition, and all specialist disciplines to avoid clashes" | Usually covered in scope — respond: "Noted — confirmed." |

## CRS Response Pattern

| Comment Status | Response Text |
|---------------|--------------|
| Already covered | "Noted. This requirement is already addressed in section [X] of the submitted SOW which states [quote relevant text]." |
| Needs confirmation | "Confirmed. [Sub] commits to [requirement] as part of their scope of services." |
| Needs new deliverable | "Acknowledged. A dated action plan / commitment to attend meetings will be added to the SOW and confirmed by [sub]." |
| Not sub's scope | "Noted. This requirement is addressed by [other sub/contractor] per the project scope split." |

## When to involve the sub vs handle internally

| Scenario | Action |
|----------|--------|
| Comment maps to existing scope text | Handle in CRS internally — no sub involvement needed |
| Comment requires sub commitment (action plan, meetings) | Ask sub to confirm in writing, include their response in CRS |
| Comment requires SOW amendment | Prepare amendment, get sub buy-in, submit revised SOW to CG |
| Comment is about another sub's scope | Note in CRS that it's assigned to the correct party; no action from this sub |

## Pitfalls

- **Don't assume Code B means full approval** — it means approved WITH comments. The comments must be closed via CRS before the SOW is contractually final.
- **Don't ask the sub to respond to standard boilerplate** — comments 2, 5, 6, 7 are always on KSA CG response forms. Handle them internally.
- **Don't conflate CG comments with scope gaps** — a comment saying "shall cover all engineering work" doesn't mean the scope was missing; it's CG's standard opening statement. Verify against actual scope sections before flagging.
- **Comments 3 and 8 are the only ones that typically require sub action** — the rest map to existing scope. Focus CRS energy there.
- **If the scope split has changed since submission** (e.g., BIM moved to another sub, shop dwgs moved to contractor), this must be reflected in the CRS response and potentially a revised SOW re-submission.

## Worked Example: ZNA Studio Lighting Designer (Aseer Museum, June 2026)

| CG Comment | In ZNA Scope? | CRS Action |
|---|---|---|
| 1 — Cover all lighting systems/spaces | ✅ Mostly — missing explicit "occupancy/daylighting sensors" and "interior lighting" as line items | Map to ZNA SOW §4; add missing line items |
| 2 — Full contractual compliance | ✅ Implied (§2 lists ER, SOW, MoC reqs) | "Noted — confirmed" |
| 3 — Dated action plan | ❌ Not in original scope | Request ZNA to provide |
| 4 — Coordinate with Nissan Richard | ✅ Covered (§4 lists all coordination parties) | Add Nissan Richard by name to CRS response |
| 5 — Comply with DMP | ✅ Covered (§2 lists BEP/MIDP/Aconex) | "Noted — confirmed" |
| 6 — All tasks reasonably required | ✅ Standard | "Noted — confirmed" |
| 7 — Full coordination | ✅ Covered (§4) | "Noted — confirmed" |
| 8 — Meetings/site visits | ❌ Not explicitly stated | Request ZNA to confirm commitment |
| 9 — Coordinate disciplines | ✅ Covered (§4) | "Noted — confirmed" |

Result: **7 of 9 comments handled internally.** Only comments 3 and 8 needed ZNA confirmation.
